# ERP Ops Toolkit

ERP 实施、运维与技术支持常用脚本工具箱。项目用于沉淀企业 ERP / 进销存 / B/S 管理系统中常见的前端误操作拦截、表单校验、数据库状态修正、库存异常处理等实践模板。

> 说明：本仓库中的脚本均为学习与演示模板，不应在生产环境中直接复制执行。涉及数据库写操作时，必须先备份、先查询确认、再在测试环境验证，最后由有权限人员按变更流程执行。

## 项目定位

这个仓库重点展示三类能力：

- 前端基础排查与修复：能通过 JavaScript 处理重复提交、必填项、金额合法性等常见问题。
- 数据库安全意识：能使用事务、备份、精准 WHERE 条件、影响行数核对，降低误操作风险。
- 实施运维闭环：能把问题现象、处理步骤、回滚方案和交付文档整理清楚，方便客户现场支持和团队交接。

适用场景：

- ERP / 进销存 / 财务 / 采购 / 库存管理系统实施支持。
- 客户现场问题排查、数据核对、操作培训和服务台账沉淀。
- 初级运维、实施工程师、技术支持工程师的脚本能力展示。

## 脚本说明

### 前端脚本

| 文件 | 场景 | 价值 |
| --- | --- | --- |
| `scripts/js/anti_double_submit.js` | 用户重复点击提交按钮，造成重复单据或重复请求 | 在提交过程中锁定按钮，避免重复提交 |
| `scripts/js/form_validator.js` | 表单必填项、金额、日期、数量等基础数据不合法 | 在请求进入后端前做基础校验，减少无效请求 |

### SQL 模板

| 文件 | 场景 | 价值 |
| --- | --- | --- |
| `scripts/sql/order_status_correction.sql` | 审批流状态卡住，单据无法继续流转 | 使用事务和影响行数校验修正状态 |
| `scripts/sql/inventory_adjustment.sql` | 库存数量异常，影响出入库业务 | 先备份异常记录，再通过事务修正库存 |

## 安全原则

1. 写操作前必须先查询：先确认目标单据、商品、客户或库存记录是否唯一。
2. 写操作前必须先备份：保留修正前的数据快照，便于回滚与审计。
3. UPDATE / DELETE 必须带精准 WHERE：禁止无条件更新，禁止用模糊条件直接改生产数据。
4. 必须检查影响行数：预期 1 行就只能影响 1 行，异常时立即回滚。
5. 生产执行必须走变更流程：测试环境验证、负责人确认、低峰执行、执行后复核。
6. 处理过程必须留痕：记录问题现象、执行脚本、执行人、时间、影响范围和结果。

## 推荐使用方式

1. 在测试库或本地模拟库中验证 SQL 模板。
2. 将脚本中的示例表名、字段名、状态值改成实际系统定义。
3. 执行前先运行 `SELECT` 语句确认目标数据。
4. 事务内执行 `UPDATE`，检查影响行数。
5. 确认无误后提交；异常时回滚。
6. 将处理过程写入服务台账或项目问题记录。
